Anthropic 在 2026 年 4 月 30 日发布了一篇技术博客，标题叫「Lessons from building Claude Code: Prompt caching is everything」。

坦率的讲，我第一次看到这个标题的时候，第一反应不是惊讶。

而是，终于有人把这件事说清楚了。

Prompt Caching，就是提示词缓存。听起来是个很底层的技术细节，但在 Claude Code 团队眼里，它不是优化项，是生存项。

Claude Code 团队把缓存命中率当成 SEV 级别来监控，低了就算事故。几个百分点的缓存命中率变化，会显著影响成本和延迟。

为什么？

因为 Claude Code 是一个长期运行的 Agent 产品。每次对话可能有几十轮、几百轮交互。如果每一轮都从头计算所有 token，成本会高到无法承受。

Prompt Caching 让 Claude Code 可以复用上一轮的计算结果，大幅降低延迟和成本。

但这件事没听起来那么简单。Anthropic 在博客里分享了一些反直觉的经验，有些甚至违反了我们平时的工程直觉。

## 01 什么是 Prompt Caching

先说一个比喻。

想象你在写一份很长的报告。你写了一章，保存了。第二天继续写，你不需要从头开始写，你只需要接着上一章往下写。

Prompt Caching 就是这个道理。

Claude 的 API 通过前缀匹配来实现缓存。所以呢，它会缓存从请求开头到每个 `cache_control` 断点之间的所有内容。也就是说，你放在请求前面的内容，越容易被缓存。

Claude Code 的系统提示词结构是这样的：

1. **静态系统提示词** 和工具（全局缓存）
2. **CLAUDE.md**（项目级别缓存）
3. **会话上下文**（会话级别缓存）
4. **对话消息**（不缓存，每轮增长）

这种布局让尽可能多的会话共享缓存命中。

![Claude Code 系统提示词布局](images/prompt-layout.png)

_Claude Code 的系统提示词分层结构：从全局缓存的基础指令到每轮增长的对话消息。_

但这种方式出奇地脆弱。Claude Code 团队之前就因为这个出过问题，比如在静态系统提示词里插入了详细的时间戳，或者非确定性排序了工具定义，或者更新了工具参数。

只要前缀变了，缓存就全废了。

## 02 五条反直觉经验

### 用消息代替系统提示词更新

系统提示词变了，缓存就没了。但有时候信息确实需要更新，比如用户改了一个文件，或者时间变了。

直觉做法是更新系统提示词。但 Claude Code 团队的做法是在下一条用户消息或工具结果里加一个 `<system-reminder>` 标签，把更新信息传进去。

这样系统提示词不变，缓存还在。

### 不要中途切换模型

这个反直觉程度最高。

Prompt 缓存是按模型隔离的。如果你在用 Opus 进行一个 10 万 token 的对话，突然想切换到一个简单的 Haiku 模型来回答一个小问题，反而会更贵。

因为你要为 Haiku 重新构建整个 prompt 缓存。

Claude Code 团队的解决方案是用子代理。Opus 准备一个「交接」消息，传给子代理，子代理用 Haiku 完成任务。

### 永远不要中途增减工具

中途改变工具集是破坏缓存最常见的方式之一。

直觉上，你觉得应该只给模型当前需要的工具。但因为工具定义是缓存前缀的一部分，增减一个工具，整个对话的缓存就失效了。

Claude Code 的 Plan Mode 就是围绕这个约束设计的。

### Plan Mode 的设计

直觉做法是，进入 Plan Mode 时，把工具集换成只读工具。但这会破坏缓存。

Claude Code 的做法是，始终保持所有工具在请求中，用 EnterPlanMode 和 ExitPlanMode 作为工具本身。用户切换到 Plan Mode 时，Agent 收到系统消息说明当前状态和指令，工具定义不变。

这还有个额外好处，模型可以自己决定进入 Plan Mode，不需要缓存断裂。

### 工具搜索，延迟加载而非移除

Claude Code 可能加载了几十个 MCP 工具。全部放在每个请求里会很贵，但中途移除又会破坏缓存。

解决方案是 `defer_loading`。不移除工具，而是发送轻量占位符（只有工具名，加上 `defer_loading: true`），模型可以通过工具搜索「发现」它们。完整的工具 schema 只在模型选择时才加载。

这样缓存前缀保持稳定，因为相同的占位符始终以相同的顺序存在。

## 03 Compaction 的安全分叉

当上下文窗口满了，Claude Code 需要压缩对话。

最简单的做法是发一个单独的 API 调用，用「总结这段对话」的系统提示词，不带工具。

但这正是成本陷阱所在。

Prompt 缓存只在请求前缀与已缓存内容逐字节匹配时生效。你的主对话用一个系统提示词和工具集缓存，压缩调用用另一个系统提示词和零工具，前缀从第一个 token 就分叉了，缓存完全不生效。

你要为整个对话支付完整的、未缓存的输入费率。对话越长，这一轮调用越贵。

### 解决方案：缓存安全分叉

Claude Code 的做法是，压缩时使用和父对话完全相同的系统提示词、用户上下文、系统上下文和工具定义。在父对话消息前面加上，然后在末尾追加压缩提示作为新的用户消息。

从 API 的角度看，这个请求和父对话的最后一个请求几乎一样，前缀相同，工具相同，历史相同，缓存前缀被复用。新增的 token 只有压缩提示本身。

但这会需要预留一个「压缩缓冲区」，确保上下文窗口有足够空间容纳压缩消息和摘要输出 token。

Anthropic 已经将这些经验内置到了 API 的 Compaction 功能中，支持 beta header `compact-2026-01-12`。

![Compaction 安全分叉流程](images/compaction-fork.png)

_Compaction 安全分叉：使用与父对话相同的系统提示词和工具集，复用缓存前缀，成本降至 1/10。_

## 04 五条核心教训

博客最后总结了五条模式：

1. **Prompt Caching 是前缀匹配**。前缀任何地方的任何变化都会使之后的所有内容失效。围绕这个约束设计整个系统。排序对了，大部分缓存优化就免费获得了。

2. **用消息代替系统提示词变更**。进入 Plan Mode、更新日期等操作，插入到对话消息中，而不是修改系统提示词。

3. **不要在对话中途改变工具或模型**。用工具来建模状态转换，而不是改变工具集。延迟加载工具，而不是移除工具。

4. **像监控正常运行时间一样监控缓存命中率**。Claude Code 团队对缓存断裂发出警报，并将其视为事故。几个百分点的缓存命中率变化会显著影响成本和延迟。

5. **分叉操作需要共享父级前缀**。如果需要运行并行计算（压缩、摘要、技能执行），使用相同的缓存安全参数，这样你就能复用父级的前缀缓存。

## 05 写在最后

Prompt Caching 这件事，表面看是一个 API 优化技巧，但往深了想，它触及了一个更根本的问题。

AI Agent 不是一个简单的问答系统。它是一个有状态的、长期运行的、多轮交互的程序。在这个程序里，每一轮交互都要重新计算所有上下文，成本会高到无法承受。

Prompt Caching 让 Agent 可以「记住」上一轮的计算结果，而不是每次都从头开始。

但这件事没听起来那么简单。它要求你：

- 把静态内容放在前面，动态内容放在后面
- 用消息传递变化，而不是修改系统提示词
- 保持工具集不变，用状态转换工具代替增减工具
- 分叉操作复用父级前缀，而不是另起炉灶

这不是一个能靠简单工具解决的问题。因为一旦你开始构建长期运行的 Agent，你就需要处理：

- 系统提示词的版本管理
- 工具集的生命周期
- 上下文窗口的压缩策略
- 缓存命中率的监控和告警

Claude Code 团队把这些经验公开分享出来，是一件很有价值的事。

如果你是 AI Agent 的开发者，我的建议是，别等成本爆了再想缓存的事。从第一天就把 Prompt Caching 纳入你的架构设计。

多花一点点时间，自己去研究一下。

毕竟，你的 Agent 可能现在就等着某一次缓存命中，来省下那几美分。
